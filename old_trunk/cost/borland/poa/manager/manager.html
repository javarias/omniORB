Manager Example

The example has 2 interfaces.  MyPOAManager is used to simulate the behavior of a POA manager by implementing all the operations of the POA manager and direct the calls to the actual POA manager.  Counter is a dummy object which keeps incrementing the number of times this object a request has been invoked on this object.

This example involves 3 parties, the server that creates the poas for the MyPOAManager and Counter objects, the client that repeatedly invokes the Counter object, and the manager that acts as the third party that controls the flow of the requests made on Counter by client.

The purpose of the example is to show how requests on objects associated with a poa can be kept in holding, active, discarding and inactivate states.  The user can randomly stop the flow of request on a particular object by putting the poa manager of the poa associated with that object in holding state.  It can also resume the flow by putting it in active state.  Under discarding state, all requests on the object will be discarded and client will receive a TRANSIENT exception when appropriate client side policies are adopted.  Requests will be resumed once the state is set to active.  Similarly, under inactive state, all requests will be rejected and client will receive an OBJ_ADAPTER exception.  The inactive state is permanent once entered.

Note: Client must set the appropriate client side policies in order to receive the TRANSIENT exception when requests are discarded.


From this example, you will learn how to:
  • Go from one poa state to another.
  • Use a third party controller to control request flow.
  • Control the flow of client request by discarding requests when overflooded.
  • Totally inactive a POA and reject all further requests.
  • Set up the client side policy to receive TRANSIENT exception.
